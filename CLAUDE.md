# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Sufusku — mobile-first Sudoku helper. User enters their own puzzle into a blank 9x9 grid (by hand or by scanning a printed puzzle with the camera); app tracks candidates (pencil marks) and highlights conflicts in real-time. Board state persists across reloads via `localStorage`.

## Commands

- `npm run dev` — start Vite dev server (HMR)
- `npm run build` — type-check (`tsc -b`) then build (`vite build`)
- `npm run lint` — run Oxlint
- `npm test` — run Vitest suite (`tests/`)
- `npm run preview` — preview production build

## Stack

React 19 + TypeScript, Vite 8, MUI 9 (Emotion), TensorFlow.js. Oxlint for linting (`.oxlintrc.json`), Vitest for tests (Node env by default; jsdom per-file via pragma), TypeScript project split into `tsconfig.app.json` / `tsconfig.node.json`. Repo is `"type": "module"` — `public/opencv/package.json` (`{"type":"commonjs"}`) exists so Node tests can `require` the UMD OpenCV build; don't delete it.

## Architecture

App logic is split across `src/sudoku/`, `src/hooks/`, and `src/components/`; `src/App.tsx` composes `Header`, `Board`, `NumberPad` (on narrow/short viewports), and `LegendDrawer`, and also owns the "solved" celebration: `isSolved` (`useMemo`, board full + `getConflictingCells` empty), a `wasSolvedRef` to detect the false→true transition (so a persisted already-solved board doesn't replay the celebration on reload), a `celebrate` counter passed to `Board` to trigger its shimmer, and a `showSolvedBanner`-driven `Snackbar`/`Alert` ("Solved!"). A second `useEffect` re-increments `celebrate` on a `setInterval` for as long as `isSolved` stays true, so the shimmer repeats every 4s until the board is edited, cleared, rescanned, or replaced by a new generated puzzle.

- `src/sudoku/coords.ts` — `getCellCoords(index)` converts a flat index to `{ row, col, box }` — the box formula is `Math.floor(row/3)*3 + Math.floor(col/3)`. This coordinate mapping is the basis for every row/column/box constraint check.
- `src/sudoku/logic.ts` — pure functions over a `Board` (`Array<number | null>` of length 81):
  - `getConflictingCells(board)` — O(n²) scan over all cell pairs sharing a row/col/box with equal values; returns a `Set` of conflicting indices, recomputed on every render.
  - `getCandidates(board, index)` — computes the pencil-mark set for an empty cell by removing values already present among row/col/box peers (local row/col/box filter only, not a full solve).
- `src/sudoku/solver.ts` — `hasSolution(board, budget)`: backtracking search with the minimum-remaining-values (MRV) heuristic (branches on the empty cell with fewest candidates first) and first-solution early exit, checking whether a board has *any* valid completion. `budget` is a shared mutable `{ remaining: number }` counter decremented once per node visited; returns `'unknown'` the instant it hits zero, so callers can bound worst-case work regardless of board emptiness. `DEFAULT_SOLVE_BUDGET` is 50,000 nodes — in practice a blank-board check finishes in single-digit milliseconds, so this is a safety net, not a real limiter.
- `src/sudoku/hint.ts` — the "Hint" feature's cell-finder, in two tiers: `findNakedSingle(board)` returns the first empty cell whose `getCandidates` set has size 1 (the same cells the board already highlights green); `findFullScanHint(board, budget?)` scans empty cells for the first one whose *true* candidate count (checked via `hasSolution`, one shared budget across the whole scan) is 1, even when its local candidate set is larger — a "hidden single" the row/col/box filter alone can't see. `findHintCell(board)` composes them: `findNakedSingle(board) ?? findFullScanHint(board)`. All three return `HintResult = { index: number; value: number } | null`.
- `src/sudoku/generate.ts` — the "New puzzle" feature. `generateSolvedBoard()`: randomized backtracking fill (MRV-ordered, like `solver.ts`, but mutates one shared `board` array in place with explicit set/backtrack instead of copying per node, since generation visits far more nodes than a single solve) — produces a complete, valid, randomly-shuffled grid. `generatePuzzle(difficulty)`: starts from a solved grid and "digs holes" in random cell order, keeping a removal only when `hasUniqueSolution` confirms the board still has exactly one solution (checked via `countSolutions`, a budget-bounded search capped at finding 2 solutions — proving non-uniqueness is cheap since it can stop at the 2nd, proving uniqueness costs the full search). Budget exhaustion is treated as "not confirmed unique," so digging always errs toward leaving a clue in rather than risking a multi-solution puzzle — meaning the actual clue count can land somewhat above the `CLUE_TARGETS` for a difficulty (easy 40 / medium 32 / hard 26) if further removal can't be proven safe within budget. Benchmarked at 1–40ms per call, so the UI calls it synchronously with no loading state.
- `src/hooks/useSudokuBoard.ts` — owns `board`, `selectedCell`, undo/redo `history`/`future` stacks, `pendingHint`, and `givenCells` (a `Set<number>` of cells filled by the last accepted scan). `setCellValue` and `clearBoard` both funnel through `applyBoard`, which pushes the pre-change board onto `history`, clears `future` (so any new action invalidates redo), and clears `pendingHint` — this single choke point is also how a stale Hint highlight gets invalidated by any edit. `setCellValue` additionally no-ops when `selectedCell` is in `givenCells`, making givens read-only. `undo`/`redo` also clear `pendingHint`. `setCellValueAt(index, value)` writes a specific cell directly, independent of `selectedCell` — used by the Hint reveal step, since the hinted cell may no longer be selected by the time the user taps reveal. `acceptScan(newBoard)` is the scan-accept path: it wipes `history`/`future` (a scan is a hard undo boundary — you can't undo past it, only through edits made after it), sets the board, and marks every non-null cell as given. `clearBoard` also empties `givenCells`. The keyboard `useEffect` on `window` `keydown` handles: digits 1-9 set value, Backspace/Delete/0 clear, arrow keys move `selectedCell` by row/col (clamped at grid edges, no wraparound), and `Ctrl/Cmd+Z` / `Ctrl/Cmd+Shift+Z` / `Ctrl/Cmd+Y` for undo/redo. `board` and `givenCells` are each lazily initialized from `localStorage` (keys `sufusku-board` and `sufusku-given-cells`, validated shape, falling back to a blank board / empty set) and persisted on every change via their own `useEffect`. Returns `{ board, selectedCell, setSelectedCell, setCellValue, setCellValueAt, setBoard (alias for the undoable `applyBoard`), acceptScan, clearBoard, undo, redo, canUndo, canRedo, pendingHint, setPendingHint, givenCells }`.
- `src/components/Board.tsx` — renders the 9x9 grid; `getCellStyling(...)` (local to this file) derives background/border/outline per cell from priority order: conflict (red) > pending-hint (amber) > single-remaining-candidate (green) > given/locked (slate) > selection (indigo bg / cyan outline) > shares-value-with-selected (cyan tint) > same row/col/box as selected (faint indigo tint). Takes `hintCell: number | null` (the pending hint's index, or `null`) and `givenCells: Set<number>` props to drive the amber and given tiers; the given background wins over selection/related/shared tints (a selected given cell still gets the cyan outline, just not the indigo selection fill). Delegates per-cell rendering to `Cell`. Sizing has three cases — see Layout breakpoints below. Also takes a `celebrate: number` prop (a counter, not a boolean — incrementing it, even to the same visual effect, is what re-triggers the animation from `App.tsx`'s repeat interval): a `useEffect` keyed on `celebrate` flips local `shimmering` state on for `SHIMMER_DURATION_MS` (1100ms), during which a `::after` pseudo-element sweeps a diagonal indigo/cyan gradient across the whole board (`zIndex: 20`, `pointerEvents: 'none'`) via the `sufusku-shimmer-sweep` keyframe (defined globally in `theme.ts`, not here, since `sx` can't declare `@keyframes` inline).
- `src/components/Cell.tsx` — presentational: renders either the resolved value or delegates to `CandidateRow` for one cell, given precomputed styling/candidates as props. `isGiven` selects the digit color (`board.givenText`, muted) over the default `secondary.main` (cyan) used for player-entered values; conflict red still wins over both.
- `src/components/CandidateRow.tsx` — the pencil-mark grid for an empty cell: a 3x3 grid filling the cell (`position: absolute; inset: 0`), digit `n` at row `floor((n-1)/3)`, col `(n-1)%3` (standard pencil-mark layout, 1 top-left...9 bottom-right). Keeps all 9 digit slots in the DOM at all times (eliminated candidates render `color: transparent` rather than being removed) to prevent layout shift. Rendered at reduced opacity (0.35) so it stays visually secondary to solved-cell values.
- `src/components/Header.tsx` — `AppBar`/`Toolbar` with the "Sufusku" title in Permanent Marker font (loaded via Google Fonts link in `index.html`), a `ThemeToggle` (sun/moon `IconButton` driven by `useColorScheme`'s `mode`/`systemMode`/`setMode`, toggling between `'dark'` and `'light'`), undo/redo `IconButton`s, and a `HeaderMenu`. Becomes a vertical strip on the left in landscape (`@media (max-height: 599.95px) and (orientation: landscape)`), with the title rotated -90deg and vertically centered below the icon group; the icon group's landscape order is menu (top), theme toggle, undo, then redo, via `order` overrides scoped to the landscape query only — portrait/desktop keep DOM order (undo, redo, toggle, menu).
- `src/components/HeaderMenu.tsx` — kebab-icon `IconButton` + MUI `Menu` with six items: "Solve cell" (disabled unless the selected cell is empty; runs `hasSolution` per locally-plausible candidate against a shared budget, then: 0 valid → "No valid value for this cell.", 1 → auto-fills via `onSolveCell`, 2-3 → lists them, >3 → "Too many possible values (N) — not a useful hint.", budget exhausted → "This cell is too complex to solve quickly."); "Hint"/"Reveal hint" (disabled when the board has no empty cells) — a two-tap flow: tap 1 calls `findHintCell(board)`, selects and highlights the result via `onSelectCell`/`onSetPendingHint` (or shows "No hint available right now." if `null`), flipping the label to "Reveal hint"; tap 2 calls `onRevealHint(pendingHint.index, pendingHint.value)` (wired to `setCellValueAt`) to fill in the value, independent of whatever `selectedCell` is by then; "Scan puzzle" (mounts the lazy-loaded `ScanDialog` — the entire scan stack loads only on first use, and its accept path is wired to `acceptScan`, not the plain undoable `setBoard`); "New puzzle" (opens `NewPuzzleDialog`, wired to the same `acceptScan` as scanning — a generated puzzle's clues become givens and it's the same hard undo boundary); "Clear all" (calls `clearBoard`); "How to use" (opens `HelpDialog`). Feedback messages render via a MUI `Snackbar`/`Alert` wrapped in `Portal` — **required**, not decorative: the `AppBar`'s `backdropFilter` establishes a CSS containing block for `position: fixed` descendants, so without the `Portal` the Snackbar renders trapped inside the 49px-tall AppBar box instead of the viewport.
- `src/components/NewPuzzleDialog.tsx` — small `Dialog` with three buttons (Easy/Medium/Hard); tapping one calls `generatePuzzle(difficulty)` synchronously and hands the board to `onGenerate` (App's `acceptScan`), then closes. No loading state — generation is fast enough (see `generate.ts`) to not need one.
- `src/components/ScanDialog.tsx` — full-screen scan flow: live `getUserMedia` viewfinder samples frames at ~10 fps (downscaled to 320px wide), auto-captures once `findGridQuad` returns a stable quad for 5 consecutive frames, then warps + classifies and moves to the review phase. A gallery icon button (top-left of the viewfinder, always present — not just on camera failure) opens the same hidden file `<input>`/`handleFile` path used by the camera-unavailable fallback screen, so picking a photo instead of scanning live is always one tap away regardless of camera state. Phases: `viewfinder` → `processing` → `review`.
- `src/components/ScanReview.tsx` — always-shown review grid before accepting a scan. A cell needs review (amber `warning.main` background) when its confidence is below 0.9 or it conflicts with another cell; tapping any cell opens a 1-9/Clear picker dialog, and editing clears the low-confidence flag (conflicts stay flagged until resolved). "Accept" hands the board up (`onScanAccept`, wired to `acceptScan` — the scanned cells become read-only givens and the acceptance is not itself undoable); "Retake" returns to the viewfinder.
- `src/components/CellLegend.tsx` — single source of truth for the 5 semantic cell states (Conflict/Given/Solvable/Hint/Selected), consumed by both `LegendDrawer` and `HelpDialog` so they can't drift apart. `getCellLegendItems(b: BoardPalette)` returns `{ label, description, preview }[]`; `preview` is a real-cell-styled mini example, not an abstract color swatch — `MiniCell` (30×30px box reproducing `Cell.tsx`'s border/bg/digit language; `candidate` prop positions a digit at the same row `⌊(n-1)/3⌋`, col `(n-1)%3` a real single-candidate cell's `CandidateRow` would — via a lightweight positioned `Box`, not `CandidateRow` itself, since `CandidateRow`'s `Typography` line-height is tuned for real (larger) board cells and overflows this 30px box) or `MiniConflictPair` (two adjacent mini cells sharing a border, mimicking how a real conflict reads on the board).
- `src/components/HelpDialog.tsx` — scrollable "How to use" dialog (statically imported, it's tiny): sections for entering numbers (incl. that given cells — from a scan or a generated puzzle — are read-only and dim the number pad's Clear button same as an empty cell), cell colors (renders `getCellLegendItems` as a list of real-cell previews, same as `LegendDrawer` — no separate "see the Legend bar" pointer needed since this section already shows the full list), the scan workflow (incl. the always-available gallery-photo option), generating a puzzle, fixing amber (low-confidence) scan cells — this one keeps its own local `Swatch` dot since it's marking a generic MUI `warning.main` color, not one of the board's semantic cell states, undo/redo (incl. that accepting a scan or generating a puzzle is a hard undo boundary), Solve cell/Hint/Clear all, and the solved-board shimmer/banner.
- `src/components/LegendDrawer.tsx` — persistent, collapsed-by-default bar rendering `getCellLegendItems`. Not modal — no backdrop, doesn't block board interaction. Bottom-anchored full-width bar in portrait/desktop (tap toggles an inline expand below the collapsed handle); in mobile landscape (`landscapeQuery`) it instead becomes a right-edge vertical strip (rotated "Legend" label, chevron toggle) that expands *leftward* over the board, capped at `33vw`, so it doesn't compete with the tight height budget there. `App.tsx` reserves space for the collapsed bar via `pb: '32px'` (portrait/desktop) or `pr: '32px'` (landscape) on its outer container.
- `src/components/NumberPad.tsx` — tap-to-fill grid shown on portrait phones (`max-width: 599.95px`), landscape phones (`max-height: 599.95px`), and desktop (`min-width: 600px and min-height: 600px`, to the right of the board); returns null when no cell is selected, so it only appears once a cell is picked. Non-candidate numbers render dimmed but stay clickable (tapping always overwrites, matching keyboard behavior) — the hook layer (`setCellValue`) is what actually blocks writes to a given cell, this component doesn't special-case the digit buttons. Layout differs by breakpoint: landscape-mobile and desktop (`maxWidth: 180px` / `220px`) use a 3-column grid with **Clear** (`onSelect(null)`) as a full-width row below the 3x3, matching the board's existing 260px/280px reserved-space formulas so pad and board never overlap. Portrait mobile (`maxWidth: 240px`, wider since it doesn't share a row with the board) instead uses a 4-column grid (`gridTemplateColumns: 'repeat(3, 1fr) 0.7fr'`) with Clear as a narrow 4th-column button spanning all 3 rows, label rotated 90° (`writingMode: vertical-rl`) — this keeps the pad shorter specifically where it matters: portrait stacks the pad above the in-flow `LegendDrawer`, so a shorter pad means less risk of needing to scroll to see it. Disabled/dimmed when the selected cell is already empty *or* is a given (`givenCells` prop) — same visual treatment for both. `App.tsx` switches the content box to a row (board left, pad right, vertically centered) under the desktop and landscape-mobile queries.
- `src/theme.ts` — dual `colorSchemes` (dark, light) via `createTheme({ cssVariables: { colorSchemeSelector: 'class' }, colorSchemes: {...} })`; components read colors through `theme.vars` CSS variables, so switching modes is pure CSS. The dark palette is unchanged from the original single-theme values (indigo `#6366f1` primary, cyan `#06b6d4` secondary, deep navy background); light is a new slate-based palette (`#f8fafc`/`#ffffff` backgrounds, `#0f172a` text). A custom `palette.board` token slot (`BoardPalette` interface, module-augmented onto `Palette`/`PaletteOptions`) holds every board-specific color (grid lines, conflict/single/selection/tint/given backgrounds, given digit text color, NumberPad border, scrollbar thumb) per scheme — components must read these tokens and never hardcode board colors. Scrollbar styles live in `MuiCssBaseline.styleOverrides`, referencing `var(--mui-palette-board-frame)` etc. The same `styleOverrides` object also declares the global `@keyframes sufusku-shimmer-sweep` used by `Board.tsx`'s solved-celebration effect — global because `sx` props can't declare keyframes inline — plus a `@media (prefers-reduced-motion: reduce)` override that redefines the same keyframe as a no-op translate, so the shimmer respects the setting without any component-level motion check. Exported via `responsiveFontSizes`.
- `src/main.tsx` — React root + `ThemeProvider` wiring; `defaultMode="system"` makes the app follow the OS color scheme until the user overrides it via `ThemeToggle` (persisted to `localStorage` key `mui-mode` by MUI itself).

## Scan pipeline (`src/scan/`)

On-device only; every module is DOM-free except `opencv.ts`, so the whole pipeline runs (and is tested) in Node.

- `src/scan/opencv.ts` — script-tag loader for the trimmed OpenCV.js wasm build (~2 MB) served from `public/opencv/`; deliberately not bundled. Caches the load promise, resets it on failure.
- `src/scan/detect.ts` — `findGridQuad(cv, image)`: adaptive threshold + contours, returns the largest convex quadrilateral covering >30% of the frame (or null); `warpGrid` perspective-warps that quad to a 450×450 grid.
- `src/scan/classify.ts` — pure pixel pipeline over `RawImage` (structural subset of `ImageData`). `findDigitBlob` locates each cell's digit by connected-component search with ±8px slack past the cell (imperfect warps shift digits off-center — hard 50px slices caused real misreads); `isEmptyCell` and `prepareDigit` **both** gate on this same blob search so the empty gate and the classifier can never disagree. `prepareDigit` downsamples a blob-centered square window to the 28×28 CNN input, normalized `(255 - gray) / 255` — MUST match `tools/train-digits/train.py`. `classifyGrid` batches all non-empty cells through the model and returns `{ board, confidence }`.
- `src/scan/model.ts` — loads the TF.js Layers model from `public/models/digits/` (~232 KB, 9 softmax classes, class d = digit d+1) and wraps it behind a tensor-free `DigitModel.predict(batch, count)` interface.
- `tools/train-digits/` — Python training pipeline for the digit CNN (synthetic renders with camera-domain augmentation: gray backgrounds, blur, stroke-width variation). Retrain: `cd tools/train-digits && .venv/bin/python train.py` (uv-managed Python 3.11 venv). `tools/opencv-build/` — Docker recipe for the trimmed OpenCV.js build.

## Tests (`tests/`)

Vitest. `realScan.test.ts` runs golden fixtures: a real photo (`tests/fixtures/real-N.png`) plus its expected 81-char board string (`real-N.txt`, `0` = empty, row-major) must match exactly — add new photos to the `it.each` list. Real-photo scanner bugs are debugged offline in Node this way, never in the live camera UI. `goldenScan.test.ts` runs the synthetic fixture end to end; `classify.test.ts` / `detect.test.ts` / `opencv.test.ts` cover the pipeline pieces on synthetic images; `useSudokuBoard.test.ts` uses jsdom via per-file pragma. `generate.test.ts` checks `generateSolvedBoard`/`generatePuzzle`: full+conflict-free, randomized across calls, every difficulty's puzzle is solvable and has fewer than 81 clues, and (over a few randomized trials, since digging can stop early) hard puzzles end up with clue counts ≤ easy puzzles'.

## Layout breakpoints

Two custom media queries recur across `App.tsx`, `Board.tsx`, `Header.tsx`, and `NumberPad.tsx` — always use the exact same query strings (not MUI's `sm`/`xs` breakpoint shorthand) to avoid cascade-order conflicts between width- and height-based rules:
- Portrait mobile: `@media (max-width: 599.95px) and (orientation: portrait)`
- Landscape mobile: `@media (max-height: 599.95px) and (orientation: landscape)` — in this mode the header collapses to a 56px-wide left strip and the board is sized via `min(calc(100vh - 16px), calc(100vw - 280px))` (applied identically to width and height — don't reuse a `%`-based expression across both properties, since `%` resolves against different bases per axis; width-relative `%` and height-relative `%` mean different things, which caused a real bug here once).
- Desktop: `@media (min-width: 600px) and (min-height: 600px)` — board fills available height via `min(calc(100vh - 113px), calc(100vw - 260px))` (81px header + 32px reserved for `LegendDrawer`'s collapsed bottom bar, matching `App.tsx`'s `pb: '32px'` on this breakpoint) instead of the plain `@media (min-width: 600px)` rule's fixed 460px cap (that rule still applies as the fallback for wide-but-short windows, which get the landscape-mobile treatment instead since the two queries are mutually exclusive on height).
